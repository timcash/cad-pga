Gear Hierarchy

Parent motion plus local motion gives one world transform.

This page documents the 2D gear-and-arm example brought over from the ga_cad/src_v1 plugin. It is deliberately simple, but it captures the core scene-graph idea cleanly. The current code uses a small SE(2)-style helper struct rather than full ganja.js multivectors.

Verification note

The code path here is SE(2) composition, with PGA as the conceptual analogue.

An earlier version of this page leaned too hard on the sandwich-product notation. The current demo does not literally run ganja.js motors; it composes angle-plus-translation structs in the plane. The MathJax below now reflects that code first, then shows the PGA analogue as a bridge.

Hierarchy equations

The child is evaluated in the parent frame.

$$ \begin{aligned} \phi_{\text{world}}(t) &= \phi_{\text{gear}}(t) + \phi_{\text{local}}(t) \\ t_{\text{world}}(t) &= t_{\text{gear}}(t) + R(\phi_{\text{gear}}(t))\,t_{\text{local}}(t) \\ p_{\text{world}} &= R(\phi_{\text{world}})\,p_{\text{local}} + t_{\text{world}} \end{aligned} $$

The orange arm is not given separate global coordinates. It inherits the gear motion and then contributes only its local offset and local oscillation. That is exactly what the multiply(parent, local) helper computes in the page script.

In code

The transform composition is straightforward.

function rotor(angle) {
  return { angle, tx: 0, ty: 0 };
}

function translator(x, y) {
  return { angle: 0, tx: x, ty: y };
}

function multiply(parent, local) {
  const cos = Math.cos(parent.angle);
  const sin = Math.sin(parent.angle);
  return {
    angle: parent.angle + local.angle,
    tx: parent.tx + cos * local.tx - sin * local.ty,
    ty: parent.ty + sin * local.tx + cos * local.ty
  };
}

The plugin version is a lightweight 2D rigid-transform model rather than a full 3D PGA implementation, but it preserves the same parent-local-world composition pattern.

PGA analogue

If the helper structs were replaced by PGA motors, the action would look like this.

$$ P_{\text{new}} = M\,P\,\widetilde{M} $$

That sandwich-product form is still the right conceptual bridge to the rest of the repo. It is just not the literal code path for this specific page. 

const mGear = rotor(gearAngle);
const mLocalOffset = translator(gearRadius - 0.2, 0);
const mLocalRotation = rotor(armLocalOscillation);
const mLocalArm = multiply(mLocalOffset, mLocalRotation);
const mWorldArm = multiply(mGear, mLocalArm);

const gearDrawn = baseGear.map((p) => applyMotor(mGear, p));
const armDrawn = baseArm.map((p) => applyMotor(mWorldArm, p));

Why it matters

The example is small, but the pattern is the important part.

  • CAD assemblies are built from the same idea: local parts inheriting parent transforms.
  • Robot chains and linkages use the same composition logic at larger scale.
  • A simple 2D page like this is a good bridge between rigid motion notation and more complex 3D demos.
