Mesh Cleanup

Boundary sums remember area even when the cone changes shape.

The live demo fans a polygon boundary out to a movable gauge point. The individual triangle facets change as the apex moves, but the signed area extracted from the full boundary sum stays fixed.

Core equations

The boundary is the real object of interest.

$$ \begin{aligned} \partial P &= \sum_{i=1}^{n} E_i, \qquad E_i = p_i \vee p_{i+1} \\ \Pi(o) &= o \vee \partial P \\ A(P) &= \frac{1}{2}\left\| \Pi(o) \right\|_{\infty} \end{aligned} $$

The area formula is not using one arbitrary triangulation of the polygon. It is using the oriented boundary. That is why the green value in the demo is stable while the red facet sum is not.

In code

The implementation mirrors the algebra directly.

let sumEdges = (pts[0] & pts[1]) * 0.0;
let absAreaSum = 0;

for (let i = 0; i < pts.length; i += 1) {
  const p1 = pts[i];
  const p2 = pts[(i + 1) % pts.length];
  const edge = p1 & p2;
  const triPlane = apex & edge;

  sumEdges = sumEdges + edge;
  absAreaSum += 0.5 * triPlane.Length;
}

const netAreaPlane = apex & sumEdges;
const netArea = 0.5 * netAreaPlane.Length;

In ganja.js dual 3D PGA, the join appears as &. The loop accumulates the oriented boundary first, then joins the apex to the whole sum.

What changes

The absolute cone sum is not the invariant quantity.

$$ A_{\mathrm{abs}}(o) = \frac{1}{2}\sum_{i=1}^{n}\left\| o \vee E_i \right\| $$

This is the red value in the UI. It depends on where the gauge point sits in space, because each cone face has a different Euclidean slope and height.

What stays fixed

The oriented boundary cancels the height terms.

$$ \frac{1}{2}\left\| o \vee \sum_i E_i \right\|_{\infty} = \frac{1}{2}\left\| \sum_i o \vee E_i \right\|_{\infty} $$

Moving the apex changes the decomposition. It does not change the signed content encoded by the boundary.

  • The red number answers “how large are the cone facets right now?”
  • The green number answers “what area does the oriented polygon boundary encode?”
  • The second question is the one that survives a change of gauge point.

Point model

The demo uses the corrected 3D PGA point embedding.

const point = (x, y, z) =>
  1e123 - x * 1e023 + y * 1e013 - z * 1e012;

This matches the 3D PGA point coordinates \(e_{123} + x e_{032} + y e_{013} + z e_{021}\) in ganja's basis ordering.
