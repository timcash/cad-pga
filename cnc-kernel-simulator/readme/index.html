Why motors are such a natural fit for CAM motion.

In a PGA-native CAD or CAM kernel, the tool does not need one system for orientation and another for translation. A single motor carries both, and the moving tool naturally suggests a swept volume for the cut.

Rotor
$$ \mathbf{R}(\theta) = \cos\!\left(\frac{\theta}{2}\right) + \sin\!\left(\frac{\theta}{2}\right)\mathbf{e}_{12} $$

The spinning spindle is modeled as a rotor around the tool axis. In the demo, this is the rapidly changing white endmill rotation.

Translator
$$ \mathbf{T}(d_x) = 1 - \frac{1}{2} d_x \mathbf{e}_{01} $$

The feed motion is an ideal-line translator. Here it slides the tool left and right through the stock block with a sinusoidal position function.

Motor
$$ \mathbf{M}(t) = \mathbf{T}(d_x(t))\,\mathbf{R}(\theta(t)) $$

Multiplying the translator and rotor gives one rigid-body motion object. That unified representation is the heart of the example.

Tool Action

Applying one motor to the full endmill

$$ p'(t) = \mathbf{M}(t)\,p\,\widetilde{\mathbf{M}(t)} $$

Every vertex of the fluted tool uses the same sandwich action. This is why the geometry stays rigid without separate bookkeeping for translation and orientation.

Swept Cut

The removed material is a kinematic sweep

$$ \mathcal{S} = \bigcup_t \mathbf{M}(t)\,\mathcal{T}\,\widetilde{\mathbf{M}(t)} $$

The red volume is a simple stand-in for the swept region of the moving tool. In a fuller PGA kernel, this sweep would become the shape subtracted from the stock.

Code Mapping

How the demo equations map into the JavaScript

  • rotor encodes the spindle spin using the bivector plane 1e12.
  • translator encodes feed motion along the x-axis using the ideal line 1e01.
  • motor = translator * rotor combines both motions into one rigid transform.
  • motor >>> p applies the sandwich action to each tool vertex.
CAD / CAM Takeaway

Why this is better than split pipelines

Traditional CAM stacks often mix matrices, Euler angles, and extra special-case logic. A motor-based formulation is cleaner because rigid motion composition stays algebraically uniform from 3-axis through more complex multi-axis setups.

The demo is intentionally simple, but the representation scales in a direction that is very attractive for simulation, swept volumes, and future tool-kernel experiments.
